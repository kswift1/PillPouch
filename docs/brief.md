## 한 줄 컨셉

**"오늘 먹었나? 헷갈리지 마세요." — 영양제를 먹은 기록이 명확하게 남는 앱.**

알람으로 챙겨주는 앱이 아니라, **먹었다는 사실에 대한 신뢰감**을 만드는 앱이다.

**메커니즘 한 문장:** *한국 약봉지 띠를 매일 가로로 찢어가며 영양제 복용을 기록한다.*

**컨셉 완성형:** *앱을 한 번도 열지 않아도 락스크린에 매일 봉지 띠가 자동으로 등장한다.* (Push to Start 기반)

## 핵심 가설

> **영양제를 챙겨 먹는 사람들의 진짜 페인포인트는 '먹었나 안 먹었나 헷갈리는 인지 부담'이다.**
> **명확한 시각적 증거(찢긴 약봉지)와 누적 기록을 보여주면, 그 인지 부담이 사라지고 영양제 챙김 자체가 더 지속된다.**

이 가설이 모든 디자인·기능 결정의 기준이다. 시안을 평가할 때 "예쁜가?"가 아니라 **"이 가설을 강화하는가?"** 로 본다.

### 가설이 함의하는 것

1. **인지 부담 해소가 본질** — "먹었나?" 떠올리는 1~10초의 반복이 누적되면 챙김 자체가 무너진다.
2. **체크는 비가역적 시각 변화여야 한다** — 봉지가 찢어진다. 한 번 본 순간 의심 없이 인지되어야 한다.
3. **누적 기록이 부수가 아니라 본질** — 오늘만이 아니라 어제·지난주가 한 화면에 보여야 한다.
4. **지속성이 진짜 KPI** — 누락률 감소가 아니라 30일 retention.

### 검증 가능한 명제

V1 출시 후 검증할 항목 3개:

1. **명제 1.** 사용자에게 "먹었나 헷갈렸던 경험"을 물으면 70% 이상이 "있다"고 답한다. → 외부 사용자 5명 인터뷰.
2. **명제 2.** 명확한 시각적 증거를 본 후 "먹었나?" 의문이 다시 생기는 빈도가 줄어든다. → 30일 도그푸딩 자체 관찰.
3. **명제 3.** 기록이 누적되어 보이면 영양제 복용 자체의 지속률이 올라간다. → 30일 사용 후 누락률 비교.

## 정체성 우선순위

| | 메인 (B) | 보조 (A) |
|---|---|---|
| **역할** | 기록 신뢰성을 만드는 앱 | 기록을 더 쉽게 남기는 도구 |
| **메커니즘** | 약봉지 띠 메타포 + 드래그 찢기 + 누적 기록 시각화 | 락스크린·위젯·다이나믹 아일랜드 자동 등장 + 한 탭 찢기 |
| **마케팅 메시지** | "오늘 먹었나? 헷갈리지 마세요" | "락스크린에서 찢어서 끝" |

**A는 B를 위한 도구다.** A가 메인처럼 보이면 정체성이 흐려진다.

## 사용자 시나리오

### 시나리오 1: 다음 슬롯 직전 (핵심 순간)

**12:25, 점심 식사 직전**

*"어, 아침에 영양제 먹었던가?"*

폰을 꺼내 **이미 락스크린에 떠있는 점심 봉지 미리보기**를 봄. (V1.0부터 Push to Start로 자동 등장)

다이나믹 아일랜드 탭. Live Activity 펼쳐짐. **이미 윗부분이 가로로 찢긴 아침 봉지** — 안에 캡슐이 빠져있고 윗 조각이 살짝 매달린 채. *"아 다 먹었네."* **의문 즉시 해소.**

옆의 봉인된 점심 봉지 윗부분을 손가락으로 좌→우 드래그. 찢어지는 미세 햅틱이 단계별로. 100% 도달 시 `.success`. 끝. **걸린 시간 5초.**

→ B 가설 충족: 의문 해소 + 새 기록의 명확한 증거.

### 시나리오 2: 슬롯 시간 지난 늦은 체크

**21:30, 저녁 식사 후 한참 뒤**

위젯에 **저녁 봉지가 봉인된 채 살짝 기울어진 모습**. *"아 맞다."* 봉지 윗부분 드래그 → 찢기. 늦었지만 기록은 남음.

안 먹기로 결정한 경우: 봉지 길게 누르기 → "건너뛰기". 점선 윤곽으로 "건너뜀" 상태로 표시.

### 시나리오 3: 어제/지난주 헷갈림

**다음날 09:00**

*"어제 저녁 비타민C 먹었나?"*

Today 화면 하단 "쌓인 증거" 영역에 **어제의 찢긴 띠**가 그대로 남아있음. 어제 저녁 봉지 자리가 찢어져 있는지 확인. 더 자세히는 "이번 주 보기" 진입점 → 주간 뷰.

## 성공 지표

V1은 단순한 자기 검증이면 충분.

1. **30일 retention** — 내가 30일 연속 사용 (가장 중요)
2. **인지 부담 감소** — 도그푸딩 중 "먹었나?" 헷갈리는 빈도가 주관적으로 감소
3. **외부 사용자 5명** — 30일 사용 후 검증 명제 1·2·3 인터뷰
4. **자동 Live Activity 도착률** — Push to Start 기반 V1.0 푸시 도착률 95% 이상

지표가 안 나오면 V1.1 작업 전에 가설 재검토.

### 30일 도그푸딩 정의

**도그푸딩(Dogfooding):** 메이커가 자기 앱을 매일 실생활에서 사용하는 것. 단순 테스트가 아니라 **"이 앱이 진짜 가치 있나?"** 검증.

**30일이 의미 있는 이유:**
- 첫 주: 신기해서 잘 씀 (가짜 신호)
- 둘째 주: 익숙해짐
- 셋째 주: 진짜 시험대 — 진짜 매일 쓰게 되나?
- 넷째 주: 결론 — 일상에 자리 잡았나?

V1.0 출시 후 즉시 시작. 결과로 V1.1 방향 결정.

## 시각 언어

> *Claude Design 입력값으로 그대로 사용 가능한 섹션.*

### 메인 메타포: 1일분 세로 약봉지 띠 (Daily Strip)

한국 약국에서 1일분 처방 시 받는 **절취선으로 연결된 봉지 띠**. 이 메타포가 가설을 시각화하는 핵심.

- **봉인된 띠** = 아직 안 먹은 하루
- **윗부분이 가로로 찢긴 봉지** = 먹었다는 비가역적 증거
- **윗부분이 다 찢긴 띠** = 그날의 완료 증거
- **누적된 찢긴 띠들** = 며칠/주 단위 신뢰성 있는 기록 시각화

**띠 방향: 세로** — 봉지가 위→아래로 절취선으로 연결.
**찢기 방향: 가로** — 각 봉지 윗부분을 좌→우로 드래그.

### 톤
- **결:** Things 3 + Streaks의 교집합. 따뜻한 미니멀, 정보 밀도 낮음, 여백 많음.
- **금기:** 의료적/병원적 톤, 형광 그린·레드 사인, 처방전 메타포, 알람 시계 메타포.
- **지향:** 자연광, 식물성, 다이어리 정서.

### 색 토큰 (방향성)
- 배경: 오프화이트 / 다크모드는 따뜻한 차콜 (순수 검정 X)
- 시간대 색조:
  - 아침 — 웜 옐로우 (`#F5C56B` 톤)
  - 점심 — 미드 코랄 / 살구 (`#E89A78` 톤)
  - 저녁 — 쿨 인디고 / 라벤더 (`#7B6BA8` 톤)

### 봉지 상태 5종

1. **봉인 (Sealed)** — 반투명 비닐, 윗부분에 V자 컷 자국, 안의 캡슐이 비쳐 보임
2. **활성 (Active, NOW)** — 살짝 빛나거나 그림자 강조, V자 컷이 미세하게 부각
3. **찢김 (Torn)** — **윗부분이 가로로 갈라져 있고, 캡슐이 빠져나간 자리. 구겨진 윗 조각이 살짝 매달려 있거나 떨어진 상태.** ← 가장 중요한 상태
4. **건너뜀 (Skipped)** — 점선 윤곽, 봉지는 봉인된 채로
5. **누락 (Missed, 시간 지남)** — 살짝 기울어짐 + 채도 낮음

### 캡슐 일러스트 (봉지 안 요소)

6종: 정제 / 소프트젤 / 캡슐 / 가루(스틱팩) / 액상(드롭) / 구미

찢김 상태에서는 **캡슐이 봉지에서 빠져나간 흔적**이 보여야 함.

### 하지 말아야 할 시각 결정
- ❌ 체크 = 봉지 위에 노란 동그라미 ✓ (B 가설 약화)
- ❌ 찢긴 봉지 안 캡슐이 흐릿함 ("먹었음" 명확하지 않음)
- ❌ Carousel로 슬롯 전환
- ❌ 단순 탭 체크
- ❌ "드래그해서 찢어주세요" 안내 카피 메인 화면 박기 (온보딩 1회만)
- ❌ "0% / 30% / 70% / 100%" 시퀀스 메인 화면 박기

## 핵심 인터랙션 명세

### 봉지 찢기 = 가로 드래그 (V1의 시그니처 인터랙션)

**행위 의미:** 사용자가 봉지를 찢어서 약을 먹는 실제 행위를 그대로 시뮬레이션. "체크"가 아니라 **"개봉"**.

#### 시각 단계 (드래그 진행도별)

| 진행도 | 시각 변화 | 햅틱 |
|---|---|---|
| **0%** (대기) | 봉지 봉인 상태. 윗부분 V자 컷 자국. | — |
| **0~30%** | V자 컷에서 살짝 갈라지기 시작. | `.soft` 진동 시작 |
| **30~70%** | 봉지 윗부분이 따라 찢어짐. 종이 찢는 지그재그 가장자리. | 진폭 점진 증가 (5~7회 누적) |
| **70~100%** | 윗 조각이 살짝 들리거나 떨어짐. 캡슐들이 살짝 보이거나 흘러나옴. | 진폭 강함 |
| **100% 도달** | 봉지 "찢김" 상태로 고정. 캡슐이 빠져나간 자리. | `.success` 강한 한 번 |

#### 드래그 임계값 (오탭 방지)

- **50% 미만에서 손 떼기** → 스프링 백
- **50% 이상에서 손 떼기** → 자동으로 100%까지 진행, 완료
- 50%가 "되돌릴 수 있는 마지막 지점" — B 가설의 신뢰성 보장

#### 드래그 방향
- **좌 → 우** (한국어 읽기 방향과 일치)

#### 햅틱 디자인
- 드래그 진행 거리에 비례한 짧은 진동 누적 (총 5~7회)
- 100% 도달 시 `.success` 한 번
- iOS의 `UIImpactFeedbackGenerator(.soft)` + `UINotificationFeedbackGenerator(.success)` 조합

#### 되돌리기 (Undo)
- 찢고 나서 5초 이내 화면 상단 "되돌리기" 토스트
- 토스트 탭 → 봉지 봉인 상태로 복귀
- 5초 후 토스트 사라짐 — 이후엔 영양제 화면에서 수동 수정만

### 길게 누르기 = 건너뛰기

봉지 길게 누르기 (0.5초+) → 액션 시트:
- 건너뛰기 (skipped 상태로 명시 기록)
- 시간 변경
- 취소

## 화면 구조

### Today 화면 레이아웃 (위→아래)

1. **헤더** — 날짜 + "오늘" + 우상단 진행도 도트 7개
2. **세로 약봉지 띠** (메인 영역)
   - 아침 / 점심 / 저녁 봉지 3개가 절취선으로 연결
   - 모든 봉지가 한 화면에 동시에 보임
   - 현재 슬롯 NOW 상태로 살짝 강조
3. **"쌓인 증거" 영역** (하단)
   - 어제·그제·지난주의 찢긴 띠들이 누적
   - 단품 봉지 카드 ❌, **하루치 띠 단위로** ✅
   - 요약: "이번 주 N개 봉지를 찢었어요"
4. **하단 탭바** (3탭) — 오늘 / 기록 / 영양제
5. **설정** — Today 우상단 톱니 아이콘 (탭바 X)

### 주간 뷰 (별도 화면, "기록" 탭)
- 7일 × 3슬롯 그리드
- 각 셀에 미니 봉지 아이콘 (봉인/찢김/건너뜀/누락)

## 기술 스택

### 클라이언트 (iOS App)
- **UI:** SwiftUI 네이티브 (TCA 미사용)
- **상태 관리:** `@Observable` 매크로 + `@Query` (SwiftData) + `@Environment(\.modelContext)`
  - 화면별 복잡도에 따라 패턴 선택. 단일 아키텍처 강요 X.
  - 단순 화면: `@Query`만으로 충분
  - 복잡한 화면(Today 드래그 인터랙션): `@Observable` ViewModel
  - 라벨로 부르자면 "MV + 화면별 ViewModel" 또는 "@Observable 패턴"
- **저장:** SwiftData (로컬) + CloudKit 동기화
- **실시간 표면:** ActivityKit + WidgetKit + AppIntent
- **워치:** WatchKit (Nice-to-have)
- **햅틱:** UIImpactFeedbackGenerator(.soft) + UINotificationFeedbackGenerator(.success)
- **iOS 최소 버전:** **iOS 17.2+** (Push to Start 전제)

### 백엔드 (V1.0 포함)
- **언어/프레임워크:** Vapor (Swift on Server) 또는 Node/Bun (속도 우선)
- **호스팅:** Fly.io 또는 Railway (월 ~$5)
- **DB:** Postgres (관리형) 또는 SQLite + Litestream
- **APNs:** HTTP/2 직접 호출 또는 `apns2` 류 라이브러리
- **스케줄러:** Cron 또는 큐 기반 (사용자 타임존 단위로 시간 슬롯 도달 시 푸시)

### TCA 도입 조건 (V2 이후 검토)
다음 중 2개 이상 충족 시:
1. 화면 10개 이상
2. CloudKit 충돌 해결 로직 복잡
3. 가족 공유 등 멀티 유저 상태 동기화
4. 외부 API 다발 (HealthKit, 식약처 등)
5. 테스트 커버리지 본격 필요
6. 협업자 합류

## ActivityKit 라이프사이클 (V1.0 기준)

### Live Activity 시작 흐름
- **로컬 시작:** 앱이 foreground일 때 `Activity.request()` 호출
- **자동 시작 (V1.0 핵심):** 사용자별 슬롯 시간 도달 시 서버가 PTS 푸시 → 앱이 killed/background 상태여도 락스크린에 자동 등장

### 업데이트 / 종료
- 사용자 봉지 찢기 → `activity.update()` (로컬)
- 슬롯 시간 도달 후 미체크 → 서버에서 PUT(Push to Update) 발송하여 NOW 상태 시각 변경
- 하루 종료 또는 모든 슬롯 완료 → `activity.end()`

### iOS 버전별 사실 (정확한 기준)
- **iOS 16.1:** Live Activity 도입. 앱 foreground에서만 시작 가능.
- **iOS 17.0:** Interactive Widget / AppIntent 도입.
- **iOS 17.2 (2023.12):** **Push to Start 도입.** PTS 토큰으로 앱 killed 상태에서도 푸시로 시작 가능. ← Pill Pouch V1.0 핵심 의존
- **iOS 18.0 (2024.09):** Broadcast Push (Channel) 도입. 다수 사용자 동시 업데이트용. **Pill Pouch는 사용 안 함** (사용자별 개별 토큰).

### Broadcast(Channel) 사용 안 하는 이유
- 영양제 알림은 사용자별 타임존/슬롯/영양제 구성이 모두 다름
- Channel은 "스포츠 스코어 같은 모두에게 동일한 이벤트"용
- Pill Pouch는 표준 PTS + 개별 토큰이 정답

## 백엔드 도입 시 유의사항 (V1.0)

V1.0에 백엔드 포함 결정 시 처음 부딪히는 것들. 일정에 미리 반영.

### 1. 인증서 / 프로비저닝
- APNs Auth Key (.p8) 발급, Team ID, Bundle ID, Key ID 세팅
- 처음이면 1~2일 헤맴
- Bundle ID에 대문자 들어가면 Topic Mismatch 에러 사례 있음 — 소문자 권장
- Live Activity용 push topic 형식: `{bundleID}.push-type.liveactivity`

### 2. 타임존 처리
- 사용자별 8시는 지역마다 다름. 서버가 사용자 IANA 타임존을 알고, **그에 맞춰 푸시 발송 시점 계산**
- 서머타임(DST) 전환 시점 주의
- 사용자가 여행 중일 때 타임존 변경 처리 (디바이스 토큰 업데이트와 함께)
- 서버 cron은 UTC로 돌리고, 각 사용자의 다음 슬롯 시각을 UTC로 계산해 큐에 넣는 패턴 권장

### 3. PTS 토큰 회전 로직
- PTS 토큰은 자주 바뀜. 갱신 처리 필요
- iOS는 디바이스 재시작 후 첫 앱 설치 시점에만 PTS 토큰 발급 — `didFinishLaunchingWithOptions` 또는 앱 시작 즉시 `registerPushToStart` 호출이 정석
- 토큰 변경 시 즉시 서버에 동기화 (`pushToStartTokenUpdates` AsyncSequence 구독)

### 4. 모니터링 / 디버깅 도구
- 푸시 안 갔을 때 원인 추적할 로그 파이프라인 필요
- APNs 응답 코드별 처리 (BadDeviceToken, Unregistered, TooManyProviderTokenUpdates 등)
- 푸시 도착률 메트릭 수집 (V1 성공 지표 4번)
- 간단하게 Sentry 또는 Axiom 연동 추천

### 5. iOS 18.x 토큰 수신 이슈 우회
- Apple Developer Forums에 보고된 알려진 이슈: **앱 강제 종료 후 재실행 시 PTS 토큰이 발급되지 않는 케이스** (iOS 18.x 다수 버전)
- 출처: Apple Developer Forums, "Live Activities Push-to-Start flows" 스레드 (2025년 기준 미해결)
- Mitigation 방안:
  - 앱 시작 시 PTS 토큰 유무 체크, 없으면 재구독
  - 일정 시간 토큰 안 받아지면 사용자에게 "앱을 한 번 더 열어주세요" 알림 (UX 폴백)
  - 로컬 노티를 항상 백업으로 유지 (PTS 실패 시에도 사용자가 알림은 받음)
- 완벽한 해결책 없음을 인지하고, 로컬 노티 폴백을 V1.0 필수로 유지

### 부수 유의사항
- App Review 시 푸시 사용 정당성 description 작성 필요 (거절 가능성 낮으나 준비)
- 사용자가 Live Activity 시스템 권한 끄면 PTS 동작 안 함 — 온보딩에서 명시
- 페이로드 크기 제한: ~4KB (ContentState 포함)
